# WorkoutCounter - Анализ требований и техническое решение

## 1. Исходный анализ требований

### Пользовательские требования
1. **Автоматическое определение упражнений** - приложение должно распознавать 4 типа упражнений:
   - Подтягивания (Pull-ups)
   - Отжимания (Push-ups)
   - Приседания (Squats)
   - Упражнения на пресс (Ab exercises)

2. **Автоматический подсчет повторений** - без нажатия кнопок
3. **Автоматическое определение начала/конца подходов** - система должна знать, когда пользователь начинает и заканчивает подход
4. **Таймер отдыха** - между подходами с автоматическим запуском
5. **Режим свободной руки** - минимальное взаимодействие с UI

### Дополнительные требования, выявленные в ходе анализа

#### 1. Обработка ошибок и граничные случаи
- Отказ камеры
- Недостаточная освещенность
- Быстрые движения
- Пользователь вне кадра
- Низкая уверенность в определении позы (< 70%)

#### 2. Оптимизация производительности
- Низкая задержка обработки (target < 200ms)
- Минимальное использование батареи
- Оптимальное использование памяти

#### 3. Пользовательский опыт
- Визуальная обратная связь о распознавании
- Звуковая обратная связь при завершении подхода
- Тактильная обратная связь (вибрация)
- Голосовые указания

#### 4. Приватность и безопасность
- Обработка видео локально на устройстве
- Отсутствие отправки видео на серверы
- Только сохранение статистики

#### 5. Доступность
- Поддержка голосовых команд
- Альтернативы вибрации для тактильной обратной связи
- Поддержка TalkBack для слабовидящих

## 2. Архитектурное решение

### 2.1 Компоненты системы

```
┌─────────────────────────────────────────────────────────┐
│                    UI Layer (Compose)                   │
├─────────────────────────────────────────────────────────┤
│         CameraScreen | WorkoutScreen | HistoryScreen    │
├─────────────────────────────────────────────────────────┤
│                   ViewModel Layer                        │
├─────────────────────────────────────────────────────────┤
│        WorkoutSessionManager (Vision Orchestration)      │
├─────────────────────────────────────────────────────────┤
│                    Vision Module                         │
│  ┌──────────────┬──────────────┬───────────────────┐   │
│  │ PoseDetector │ Classifier   │ RepetitionCounter │   │
│  └──────────────┴──────────────┴───────────────────┘   │
├─────────────────────────────────────────────────────────┤
│          Repository Layer + Domain Models               │
├─────────────────────────────────────────────────────────┤
│                  Room Database                          │
└─────────────────────────────────────────────────────────┘
```

### 2.2 Основные классы

#### Vision Module
- **PoseDetector**: Использует MediaPipe для определения 33 точек тела
- **ExerciseClassifier**: Анализирует позы и определяет тип упражнения
- **RepetitionCounter**: Считает повторения на основе циклов движений
- **WorkoutSessionManager**: Координирует все компоненты, управляет состоянием
- **ErrorHandler**: Обработка ошибок и граничных случаев

#### Data Layer
- **Entities**: ExerciseEntity, WorkoutEntity, SetEntity, RepEntity
- **DAOs**: ExerciseDao, WorkoutDao, SetDao, RepDao
- **Repositories**: ExerciseRepository, WorkoutRepository, SetRepository, RepRepository
- **Database**: AppDatabase с использованием Room

### 2.3 Алгоритм определения упражнений

Классификация основана на положении ключевых точек тела (MediaPipe Pose):
- **Подтягивания**: Руки над плечами, тело вертикально
- **Отжимания**: Тело горизонтально, руки согнуты/разогнуты
- **Приседания**: Ноги согнуты, спина относительно вертикальна
- **Упражнения на пресс**: Тело лежит, мышцы пресса работают

### 2.4 Алгоритм подсчета повторений

1. Вычисляются углы в суставах (локоть, колено, бедро)
2. Определяется фаза движения: REST → ASCENDING → TOP → DESCENDING → BOTTOM → REST
3. Одно повторение = полный цикл от вершины к низу и обратно
4. Требуется 3+ последовательных фрейма для подтверждения смены фазы (фильтр шума)

## 5. Улучшения архитектуры

### 5.1 Обработка ошибок
- Перехват ошибок камеры с различной типизацией
- Обнаружение низкой уверенности в распознавании (< 70%)
- Локализованные сообщения об ошибках на русском языке

### 5.2 Производительность
- Фильтрация шума через требование 3+ последовательных фреймов
- Кэширование последних координат для избежания пересчетов
- Использование Flow для реактивного обновления состояния

### 5.3 Расширяемость
- Легко добавлять новые типы упражнений через ExerciseClassifier
- Модульная архитектура позволяет заменять компоненты
- Зависимость через Hilt для управления жизненным циклом

### 5.4 Тестируемость
- Все компоненты имеют четкие интерфейсы
- Возможность использования mock-объектов
- Отделение бизнес-логики от Android-специфичного кода
