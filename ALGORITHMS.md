# WorkoutCounter - Технические детали алгоритмов

## 🧠 Система классификации упражнений

### Обзор

Система использует MediaPipe Pose для получения 33 ключевых точек тела в реальном времени и анализирует их геометрию для определения типа выполняемого упражнения.

### MediaPipe Pose - 33 Ключевые точки

```
0   Nose (нос)
1-2 Eyes (глаза)
3-4 Ears (уши)
5-6 Shoulders (плечи) - используются как 11-12
7-8 Elbows (локти)
9-10 Wrists (запястья)
11-12 Hips (бедра) - используются как 23-24
...
13-26 Upper/Lower body (туловище)
27-28 Ankles (лодыжки)
29-32 Feet (ступни)
```

### Детектирование упражнений

#### 1. PULL_UPS (Подтягивания)

**Алгоритм:**
```
1. Получить координаты ключевых точек:
   - leftWrist (9), rightWrist (10)
   - leftShoulder (11), rightShoulder (12)
   - leftHip (23), rightHip (24)

2. Проверить условия:
   a) Запястья выше плеч:
      wristsAboveShoulder = (wrist.y < shoulder.y)
   
   b) Тело вертикально:
      bodyVertical = abs(leftHip.x - rightHip.x) < 0.15f
   
   c) Обе условия должны быть истинными

3. Результат: ExerciseType.PULL_UPS
```

**Примеры координат:**
```
Позиция внизу (руки вытянуты):
- Запястья: y = 0.7
- Плечи: y = 0.5
- Результат: wrist.y > shoulder.y ❌ NOT pull-up

Позиция вверху (руки согнуты):
- Запястья: y = 0.1
- Плечи: y = 0.5
- Результат: wrist.y < shoulder.y ✅ PULL_UP detected
```

#### 2. PUSH_UPS (Отжимания)

**Алгоритм:**
```
1. Получить координаты:
   - nose (0)
   - leftWrist (9), rightWrist (10)
   - leftHip (23), rightHip (24)
   - leftElbow (7), rightElbow (8)

2. Проверить условия:
   a) Тело горизонтально:
      bodyHorizontal = abs(nose.y - avg(hip.y)) < 0.3f
   
   b) Запястья под туловищем:
      wristsUnderBody = (wrist.y > nose.y)
   
   c) Обе условия должны быть истинными

3. Результат: ExerciseType.PUSH_UPS
```

**Примеры координат:**
```
Наверху (руки вытянуты):
- Нос: y = 0.3
- Запястья: y = 0.35
- Бедра: y = 0.35
- Результат: тело горизонтально + запястья под туловищем ✅

Внизу (на полу):
- Нос: y = 0.3
- Запястья: y = 0.4
- Бедра: y = 0.35
- Результат: тело все еще горизонтально ✅
```

#### 3. SQUATS (Приседания)

**Алгоритм:**
```
1. Получить координаты:
   - leftHip (23), rightHip (24)
   - leftKnee (25), rightKnee (26)
   - leftAnkle (27), rightAnkle (28)

2. Вычислить углы в коленях:
   leftKneeBend = calculateAngle(hip, knee, ankle)
   rightKneeBend = calculateAngle(hip, knee, ankle)

3. Проверить условие:
   squatDetected = (leftKneeBend < 120° AND rightKneeBend < 120°)

4. Результат: ExerciseType.SQUATS
```

**Функция calculateAngle:**
```kotlin
fun calculateAngle(p1, p2, p3): Float {
    // Вычисляем два вектора
    vec1 = p1 - p2
    vec2 = p3 - p2
    
    // Скалярное произведение
    dotProduct = vec1 · vec2
    
    // Магнитуды векторов
    mag1 = |vec1|
    mag2 = |vec2|
    
    // Косинус угла
    cosAngle = dotProduct / (mag1 * mag2)
    
    // Угол в градусах
    angle = arccos(cosAngle) * 180 / π
    
    return angle
}
```

**Примеры координат:**
```
Полная приседание (колени согнуты 90°):
- Hip: y = 0.4
- Knee: y = 0.6
- Ankle: y = 0.8
- Угол: 90° < 120° ✅ SQUAT detected

Стоячая позиция (колени прямые 180°):
- Hip: y = 0.4
- Knee: y = 0.3
- Ankle: y = 0.2
- Угол: 180° > 120° ❌ NOT a squat
```

#### 4. AB_EXERCISES (Упражнения на пресс)

**Алгоритм:**
```
1. Получить координаты:
   - nose (0)
   - leftShoulder (11), rightShoulder (12)
   - leftHip (23), rightHip (24)

2. Проверить условия:
   a) Большое расстояние нос-бедро:
      bodyHorizontal = abs(nose.y - avg(hip.y)) > 0.3f
   
   b) Плечи ниже бедер:
      shouldersBelowHips = avg(shoulder.y) > avg(hip.y)
   
   c) Одно из условий должно быть истинным

3. Результат: ExerciseType.AB_EXERCISES
```

**Примеры координат:**
```
Расслабленная позиция (лежа):
- Нос: y = 0.8
- Плечи: y = 0.7
- Бедра: y = 0.6
- Результат: тело горизонтально ✅ AB_EXERCISE

Скручивание (сокращение):
- Нос: y = 0.6
- Плечи: y = 0.65
- Бедра: y = 0.6
- Результат: плечи не ниже бедер, расстояние минимально ✅ AB_EXERCISE
```

## 🔄 Система подсчета повторений

### Концепция

Система отслеживает фазы движения и подсчитывает полные циклы.

### Фазы движения

```
REST
  ↓
ASCENDING (движение вверх)
  ↓
TOP_POSITION (вершина)
  ↓
DESCENDING (движение вниз)
  ↓
BOTTOM_POSITION (дно)
  ↓
REST
```

### Критерии фаз для разных упражнений

#### PUSH_UPS

```
Угол локтя (calculateElbowBend):

< 60°       → BOTTOM_POSITION (максимально согнут)
60° - 140°  → ASCENDING (если угол растет) / DESCENDING (если падает)
> 140°      → TOP_POSITION (максимально разогнут)

Переход REST → TOP → BOTTOM → TOP = 1 повторение
```

#### PULL_UPS

```
Высота запястья относительно плеча:

wrist.y < shoulder.y - 0.2f  → TOP_POSITION
wrist.y > shoulder.y + 0.1f  → BOTTOM_POSITION
                               → ASCENDING / DESCENDING

Переход REST → TOP → BOTTOM → TOP = 1 повторение
```

#### SQUATS

```
Угол колена (calculateKneeBend):

< 90°        → BOTTOM_POSITION (максимально согнут)
90° - 160°   → ASCENDING (если угол растет) / DESCENDING (если падает)
> 160°       → TOP_POSITION (максимально разогнут)

Переход REST → TOP → BOTTOM → TOP = 1 повторение
```

#### AB_EXERCISES

```
Расстояние грудь-бедро (calculateChestHipDistance):

< 0.2f      → TOP_POSITION (максимально сокращено)
0.2f - 0.4f → ASCENDING (если растет) / DESCENDING (если падает)
> 0.4f      → BOTTOM_POSITION (максимально расслаблено)

Переход REST → TOP → BOTTOM → TOP = 1 повторение
```

### Фильтрация шума

**Проблема:** Случайные колебания координат могут привести к неправильному подсчету

**Решение:** Требуется 3+ последовательных фрейма для подтверждения смены фазы

```kotlin
consecutiveFramesInPhase++

if (consecutiveFramesInPhase >= framesNeededForPhaseChange) {
    // Подтвердить смену фазы
    motionPhase = newPhase
    consecutiveFramesInPhase = 0
    
    // Проверить, завершено ли повторение
    if (isRepCompleted(oldPhase, newPhase)) {
        repCount++
    }
}
```

### Точность подсчета

```
Идеальный сценарий:
- Четкие полные движения
- Хорошее освещение
- Человек в центре камеры
- Скорость выполнения умеренная (не слишком быстро)

Результат: 95%+ точность

Сложные сценарии:
- Быстрые движения (может пропустить)
- Неполные движения (не считаются)
- Рывковые движения (может считать несколько раз)
- Движения по краям кадра (низкая уверенность)

Результат: 70-90% точность
```

## 🎯 Управление сессией упражнений

### Диаграмма состояний

```
                   ┌─────────────┐
                   │    Idle     │ (начальное состояние)
                   └──────┬──────┘
                          │ startSession()
                          ↓
                   ┌─────────────┐
                   │  Detecting  │ (поиск упражнения)
                   └──────┬──────┘
                          │ обнаружено упражнение
                          ↓
                   ┌─────────────────────────┐
                   │  SetInProgress          │ (выполняется подход)
                   │  - счетчик повторений   │
                   │  - отслеживание рип     │
                   └──────┬──────────┬────────┘
                          │          │
              человек     │          │ новое упражнение
              встает       │          │
                    ┌──────┴────┐    │
                    ↓           ↓    │
            ┌──────────────┐    │    │
            │ SetCompleted │────┼────┘
            │ - начало      │    │
            │   отдыха      │    │
            └──────┬────────┘    │
                   │             │
              отдых закончен     │
                   │             │
                   ↓             ↓
            ┌────────────────────────┐
            │     Resting / Idle     │
            └────────────────────────┘
                   ↑
                   │ новое упражнение
                   │
            ┌────────────────────┐
            │  SetInProgress     │
            └────────────────────┘
```

### Переходы состояний

```kotlin
enum class SessionState {
    Idle,              // Начальное, приложение готово
    Detecting,         // Поиск упражнения
    SetInProgress,     // Активное упражнение
    Resting,           // Период отдыха
    SetCompleted       // Подход завершен
}

// Переходы:
Idle → Detecting (startSession)
Detecting → SetInProgress (обнаружено упражнение)
SetInProgress → SetCompleted (человек встал)
SetCompleted → Resting (автоматический переход)
Resting → SetInProgress (новое упражнение)
Resting → Idle (таймер отдыха истек)
SetInProgress → SetInProgress (переход на другое упражнение)
```

### Логика обнаружения конца подхода

```kotlin
fun handleSetEnd() {
    // Условия завершения подхода:
    
    1. Пользователь встает (isStanding == true)
       ↓
       → EndSet & StartRest (60 сек)
    
    2. Обнаружено другое упражнение
       ↓
       → EndSet & StartNewSet
    
    3. Потеря отслеживания (confidence < 0.5)
       ↓
       → ErrorState, SetCompleted
}
```

## 📊 Обработка ошибок и граничные случаи

### Типы ошибок

```
┌─────────────────────────────────────────────────────┐
│              WorkoutError Types                     │
├─────────────────────────────────────────────────────┤
│                                                     │
│ 1. CameraNotAvailable                              │
│    - Устройство без камеры                          │
│    - Действие: Показать ошибку, выход              │
│                                                     │
│ 2. CameraAccessDenied                              │
│    - Отсутствует permission CAMERA                  │
│    - Действие: Запросить permission                 │
│                                                     │
│ 3. InvalidPoseDetection                            │
│    - Низкая уверенность < 50%                      │
│    - Действие: Подсказать лучше открыть лицо       │
│                                                     │
│ 4. LowConfidencePose                               │
│    - Уверенность 50-70%                            │
│    - Действие: Предупредить, продолжить             │
│                                                     │
│ 5. ExerciseDetectionFailed                         │
│    - Не удается определить упражнение              │
│    - Действие: Запросить выполнять четче            │
│                                                     │
│ 6. ProcessingError                                 │
│    - Общая ошибка обработки                        │
│    - Действие: Перезагрузить сессию                │
└─────────────────────────────────────────────────────┘
```

### Граничные случаи

```
1. Быстрые движения
   - Фреймы могут пропускаться
   - Решение: Интерполяция между фреймами

2. Неполные движения
   - Фаза не достигает максимума
   - Решение: Требование минимального угла

3. Дрожание (jitter)
   - Координаты скачут вверх-вниз
   - Решение: Сглаживание через Калман-фильтр

4. Несколько людей
   - Мультипл обнаружений
   - Решение: Использование человека с наибольшей уверенностью

5. Частичное видимое тело
   - Некоторые точки не видны
   - Решение: Проверка уверенности каждой точки
```
